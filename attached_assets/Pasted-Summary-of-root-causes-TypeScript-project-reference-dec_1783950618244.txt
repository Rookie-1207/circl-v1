Summary of root causes

TypeScript project-reference / declaration mismatch: TS6305 shows api-server is importing types from built packages (lib/db and lib/api-zod) but their output .d.ts files weren't produced before api-server typechecking. Build / project-reference ordering in CI is incorrect.
Many TS7006 / TS7031 "implicitly has an 'any' type" errors in route files. These are real TypeScript errors (missing parameter typings) that need source fixes (or a deliberate tsconfig change) — they will continue to fail typecheck until fixed.
Concrete, minimal fixes

Ensure dependent packages are built before typechecking api-server Add a step in .github/workflows/ci.yml to build the workspace packages that api-server depends on (at least @workspace/db and @workspace/api-zod) before running the api-server typecheck. This resolves TS6305 by producing the referenced declaration files.

Fix the implicit-any source errors You must add proper type annotations in the reported files (routes/activities.ts, connections.ts, conversations.ts, dashboard.ts, notifications.ts, users.ts, and any middlewares). A few general patterns and examples:
Route handlers Before: router.get('/', async (req, res) => { ... })
After: import type { Request, Response, NextFunction } from 'express'

router.get('/', async (req: Request, res: Response, next: NextFunction) => { ... })

Array callbacks / map/filter/reduce Before: items.map(a => doSomething(a))
After: items.map((a: Activity) => doSomething(a))

Destructured binding elements Before: const { _s, _d } = payload
After (example with types): const { _s, _d }: { _s: string; _d: Date } = payload

Generic parameters returned by database layer Use the exported types from your db/api-zod packages where applicable, e.g.: import type { UserRecord, ActivityRecord } from '@workspace/db' items.map((a: ActivityRecord) => ...)
If you need a quick temporary bypass (not recommended long-term), you can relax the rule in tsconfig.json by setting "noImplicitAny": false in api-server/tsconfig.json — but it's better to add explicit types so the code stays correctly typed.

If you use TypeScript project references If your repository uses project references (composite builds), ensure:
Each referenced package has "composite": true and emits declarations.
CI builds referenced projects (pnpm --filter ... run build) before depending projects are typechecked. Alternatively, run a workspace-wide build: pnpm -w -r run build (but this may be slower than building only the needed packages).
Optional: workspace-wide typecheck in correct dependency order You can also change the CI to typecheck the whole workspace in order, which will enforce building references in the right order. Example:
name: Typecheck — all workspaces run: pnpm -w -r run typecheck
But this requires each package's "typecheck" script to be configured to respect project references. The targeted build-before-typecheck approach is more surgical.
